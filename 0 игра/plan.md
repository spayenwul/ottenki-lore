### Новая Архитектура: Три Столпа Генерации

Вся система должна стоять на трех столпах:

1.  **Фундамент Данных (The "What"):** Что существует в этом мире? (Ваши YAML-файлы).
2.  **Ядро Логики (The "How"):** Как эти элементы взаимодействуют и сочетаются? (Новые сервисы).
3.  **Инструменты Генерации (The "Builder"):** Как из данных и правил построить уникальный мир? (Новые генераторы).

---

### 1. Фундамент Данных: Реструктуризация `data`

Ваши YAML-файлы — это мозг всей системы. Их нужно организовать максимально логично. Предлагаю следующую структуру директории `data/`:

**`data/`**
├── **`tags_registry.yaml`**: **ЗАКОН.** Единый и единственный список ВСЕХ возможных тегов в игре. Если тега здесь нет, его не существует. Это предотвратит опечатки и обеспечит целостность.

├── **`world_anatomy.yaml`**: **КАНОН.** Неизменяемая география мира: континенты (Торакс, Хватательная Конечность) и какие *типы* регионов на них могут быть.

├── **`location_compatibility.yaml`**: **ПРАВИЛА СОСЕДСТВА.** Какие биомы могут появляться в каких регионах, какие не могут граничить друг с другом, их вес (шанс появления).

└── **`data_tables/`**: **ЭНЦИКЛОПЕДИЯ МИРА.**
    ├── `__init__.py` (для удобного импорта)
    ├── `anatomy.yaml` (Типы регионов, Биомы, Лендмарки)
    ├── `ecosystem.yaml` (Флора, Фауна, Эко-угрозы)
    ├── `inhabitants.yaml` (Расы, Фракции, Типы поселений)
    ├── `economy.yaml` (Ресурсы, Валюта, Отрасли)
    ├── `history.yaml` (Древние цивилизации, События, Мифы)
    └── `manifestations.yaml` (Типы "магии", Аномалии)

#### Ключевое изменение в YAML-файлах:

Каждая запись в "Энциклопедии" (раса, биом, предмет) должна иметь стандартизированную структуру:

```yaml
# Пример записи в data/data_tables/inhabitants.yaml
RACES:
  - id: kith # Уникальный ID
    name: Киты (Дети Кости)
    description: "Прагматичные строители..."
    
    # СТОЛП №1: Теги - кто они такие?
    tags:
      - "быт:оседлый"
      - "культура:фортификационная"
      - "отношение_к_миру:прагматизм"
      - "технологии:кость_и_хитин"
      
    # СТОЛП №2: Совместимость - где и с кем они живут?
    compatibility:
      # С какими тегами они хорошо сочетаются (увеличивает шанс их появления)
      tags:
        "рельеф:горизонтальный": 1.5 # Вес > 1 - предпочитают
        "земля:стабильная": 2.0     # Очень предпочитают
        "ресурс:есть_костехитин": 3.0 # Жизненно необходимо
      # С какими конкретно расами они дружат
      races: ["dorsals", "geodes"]
      
    # СТОЛП №3: Несовместимость - чего они избегают?
    incompatibility:
      # С какими тегами они конфликтуют (уменьшает шанс)
      tags:
        "земля:подвижная": 0.2 # Вес < 1 - избегают
        "экология:хаотичная": 0.1
      # С какими расами они враждуют
      races: ["ursines", "chimerics"]
```

**Результат:** Ваши данные теперь не просто списки, а **база знаний с правилами**, которая готова к сложной математической обработке.

### 2. Ядро Логики: Новые и Переработанные Сервисы

Сервисы — это "безмозглые" инструменты, которые работают с данными.

*   `WorldDataService`: Его роль **упрощается**. Он только загружает YAML и предоставляет удобные геттеры: `get_race('kith')`, `get_biome_data('toxic_swamp')`. Он не принимает решений, он просто подает "патроны".

*   `TagRegistryService`: Загружает `tags_registry.yaml` и отвечает на один вопрос: "Этот тег легален?".

*   **`CompatibilityService` (Новый, ключевой сервис):**
    *   Принимает на вход два объекта (например, расу и биом) или два набора тегов.
    *   Смотрит в их поля `compatibility` и `incompatibility`.
    *   На основе весов и правил выдает "оценку совместимости" (число от 0 до бесконечности). Например, `get_compatibility_score(kith, pulsating_plains)` вернет низкое число, а `get_compatibility_score(kith, spine_ridge)` — высокое.
    *   Это — математическое ядро вашей системы.

### 3. Инструменты Генерации: Создание `generators/`

Это самое главное. Мы выносим всю логику генерации в отдельную папку `generators/`. Эти скрипты — ваши "художники".

#### **Генератор Карты Региона (RimWorld-style)**

Это самый сложный и интересный генератор.

1.  **Входные данные:** Тип региона (например, "Хребет-Остов") и его базовые теги (`рельеф:вертикальный`, `ресурс:есть_костехитин`).
2.  **Шаг 1: Генерация Карт Шумов.** Скрипт создает несколько 2D-массивов (карт) с помощью **шума Перлина/Симплекса**:
    *   `elevation_map` (карта высот)
    *   `metabolism_map` (карта "температуры")
    *   `lymph_map` (карта "влажности")
3.  **Шаг 2: Определение Биомов.** Скрипт проходит по каждой клетке (гексу) карты:
    *   Смотрит на значения шумов в этой клетке (например, `height=0.8`, `temp=0.2`).
    *   На основе этих значений присваивает клетке первичные теги (например, `рельеф:высокий`, `метаболизм:холодный`).
    *   Далее он обращается к `WorldDataService`, чтобы получить список всех *потенциально возможных* биомов для "Хребта-Остова".
    *   Для каждого гекса он использует `CompatibilityService`, чтобы найти биом, теги которого **наиболее совместимы** с тегами, сгенерированными шумом. (`get_best_match(tags_from_noise, possible_biomes)`).
4.  **Результат:** У вас есть 2D-карта региона (например, 256x256), где каждый гекс — это определенный тип биома (`bone_needles`, `litho-lichen_slopes` и т.д.), расположенный логично благодаря картам шумов.

#### **Генератор Паспортов**

После того как карта сгенерирована, запускается этот генератор.

1.  **Паспорт Региона:**
    *   Анализирует всю сгенерированную карту региона.
    *   Считает, какие биомы доминируют, какие встречаются редко.
    *   Определяет общие теги для всего региона.
    *   Используя `CompatibilityService`, подбирает доминирующую расу, фракции, уникальные лендмарки, которые лучше всего подходят этому набору тегов.
2.  **Паспорт Биома:**
    *   Для каждого уникального биома на карте он создает свой паспорт.
    *   Подбирает характерную флору, фауну, ресурсы, которые совместимы с тегами этого биома.
3.  **Паспорт Локации (Point of Interest):**
    *   Внутри каждого биома он "раскидывает" несколько конкретных точек: "Логово Стрекотунов", "Заброшенная шахта Китов", "Роща Нейроцветов".
    *   Каждая точка — это тоже набор тегов, совместимый с биомом и регионом.

### Roadmap: План действий для перехода на новую архитектуру

Вот как я предлагаю реализовать этот переход.

#### **Фаза I: Фундамент (Завершение Базы Данных)**
*Цель: Создать полный, логически связанный и готовый к анализу "скелет" мира.*

1.  **Рефакторинг `data` (Немедленно):**
    *   [ ] Создать структуру папок и файлов, как описано выше (`data/tags_registry.yaml`, `data/data_tables/anatomy.yaml` и т.д.).
    *   [ ] Перенести все существующие данные из старых файлов в новые YAML.

2.  **Наполнение "Энциклопедии" (Систематическая работа):**
    *   [ ] **Приоритет №1: Расы.** Полностью заполнить `inhabitants.yaml`, детализируя все расы с тегами, правилами совместимости и несовместимости. *Это самое важное, так как мир существует для его обитателей.*
    *   [ ] **Приоритет №2: Биомы и Регионы.** Заполнить `anatomy.yaml` по тому же принципу.
    *   [ ] **Приоритет №3: Остальные разделы.** Последовательно заполнить остальные файлы.

3.  **Реализация Сервисов Ядра:**
    *   [ ] Переписать `WorldDataService` для работы с новой структурой YAML.
    *   [ ] Написать `TagRegistryService`.
    *   [ ] Написать `CompatibilityService`. Начните с простой логики (сумма/произведение весов), усложнить можно будет потом.

**Результат Фазы I:** У вас есть полный, структурированный и логически связанный набор данных и базовые сервисы для работы с ним. Старая логика генерации еще работает, но основа для новой уже заложена.

#### **Фаза II: Инструментарий (Создание Генераторов)**
*Цель: Построить набор "кистей" для рисования мира.*

1.  **Разработка "Генератора Паспортов Регионов":**
    *   [ ] Написать скрипт, который **не генерирует карту**, а просто на основе системы весов и совместимости создает один полный "паспорт" региона (от типа органоида до местных легенд).
    *   [ ] **Визуализация и Балансировка:** Сгенерировать 1000 паспортов и с помощью `Matplotlib/Seaborn` построить гистограммы. Вы увидите, какие расы и биомы выпадают слишком часто, а какие — никогда. Это позволит отбалансировать веса в YAML-файлах. *Это итеративный процесс: сгенерировал -> проанализировал -> исправил -> повторил.*

2.  **Разработка "Генератора Физической Карты":**
    *   [ ] Начать с простого: написать скрипт, который генерирует карты шумов Перлина и сохраняет их как картинки. Поиграйтесь с параметрами (масштаб, октавы).
    *   [ ] Реализовать алгоритм, который на основе карт шумов и `CompatibilityService` создает 2D-массив с ID биомов.

**Результат Фазы II:** У вас есть работающие инструменты, способные создавать логичные и разнообразные "паспорта" регионов и их физические карты.

#### **Фаза III: Творчество (Наполнение Мира и Подключение LLM)**
*Цель: Использовать созданные инструменты для "сборки" мира.*

1.  **Создание Каноничной Карты:**
    *   [ ] Запустить генераторы несколько раз, пока не получится вариант, который "ощущается" правильным. Сохранить его.
    *   [ ] Для каждого региона на этой карте сгенерировать "паспорт".

2.  **"Оживление" через LLM:**
    *   [ ] Создать промпт, который принимает на вход **структурированный паспорт** региона/биома и просит LLM:
        *   Написать связное, художественное описание.
        *   Придумать 1-2 уникальных квеста или конфликта, логично вытекающих из данных паспорта.
        *   **Предложить новый, уникальный элемент для `data_tables` (например, новый вид флоры, новый миф), который подходит этому паспорту.**

3.  **Расширение Канона:**
    *   [ ] Анализировать предложения LLM. Лучшие идеи **вручную** добавлять в ваши `data_tables`.
    *   [ ] Повторять цикл: генерация -> обработка LLM -> расширение базы данных.
