## Часть 1: Схема-документация: Архитектура тегов мира Сильгаррон

Ваша система генерации мира построена на мощном и гибком фундаменте тегов. Она состоит из трех логических уровней, которые вместе определяют всё, от ландшафта до культуры обитателей.

#### Уровень 1: Фундамент Данных — "Что существует?"

Это ваши YAML-файлы в директории `data/`. Они являются источником истины.

1.  **`data/tags_registry.yaml` — ЗАКОН.**
    *   **Назначение:** Единственный и абсолютный список ВСЕХ возможных тегов в игре. Если тега здесь нет, он не существует для системы. Это главный файл для предотвращения опечаток и обеспечения целостности данных.
    *   **Структура:** Теги разделены на иерархические и логические группы (`pillars`, `defining_tags`, `flavor_tags`). Здесь же задаются глобальные правила взаимодействия тегов (`synergies`, `conflicts`).

2.  **`data/data_tables/*.yaml` — ЭНЦИКЛОПЕДИЯ.**
    *   **Назначение:** Это база данных всех игровых сущностей: рас (`inhabitants.yaml`), биомов (`anatomy.yaml`), фракций, предметов и т.д.
    *   **Ключевой принцип:** Каждая сущность (например, раса "Киты") описывается через набор тегов из `tags_registry.yaml`. Именно эти теги определяют её сущность для игрового движка.

#### Уровень 2: Ядро Логики — "Как всё сочетается?"

Это сервисы, которые "читают" вашу энциклопедию данных и делают выводы на основе тегов.

1.  **`TagRegistryService` (`services/tag_registry_service.py`)**
    *   **Задача:** Загружает `tags_registry.yaml` и отвечает на один простой вопрос: "Этот тег легален?". Используется всеми остальными системами для валидации.

2.  **`WorldDataService` (`services/world_data_service.py`)**
    *   **Задача:** Загружает всю "Энциклопедию" (`data_tables`) и предоставляет удобный доступ к данным (`get_race('kith')`, `get_biome('pulsating_plains')`). Он просто подает "патроны" другим сервисам.

3.  **`CompatibilityService` (`services/compatibility_service.py`) — МОЗГ СИСТЕМЫ.**
    *   **Задача:** Это математическое ядро вашей игры. Он принимает две сущности (например, расу и биом), смотрит на их теги и правила совместимости (`environmental_compatibility`), сверяется с глобальными правилами из `tags_registry.yaml` и `generation_rules.yaml` и вычисляет **"оценку совместимости"** — число.
    *   **Пример:** `get_race_biome_compatibility('kith', 'toxic_swamp')` вернет очень низкое число, потому что теги Китов (`surface:stable`, `lifestyle:sedentary`) конфликтуют с тегами Токсичных топей (`surface:unstable`, `ecology:toxic`).

#### Уровень 3: Инструменты Генерации — "Как мир строится?"

Это будущие генераторы (`generators/`), которые будут использовать сервисы из Уровня 2 для создания уникального мира.

*   **Генератор Карты:**
    1.  Создает "карты шумов" (высота, влажность).
    2.  Для каждой клетки на карте определяет набор тегов (например, `terrain:elevated`, `ecology:barren`).
    3.  С помощью `CompatibilityService.find_best_match()` находит в `WorldDataService` биом, теги которого **наиболее совместимы** с тегами этой клетки.
    4.  Так, клетка за клеткой, рождается логичная и уникальная карта мира.

**Схема взаимодействия:**

```
[ Генератор ]  <-- использует --> [ CompatibilityService ]  <-- читает правила из --> [ generation_rules.yaml ]
      ^                                      ^
      | использует                           | использует
      v                                      v
[ WorldDataService ]  <-- загружает --> [ data_tables/*.yaml ]  <-- используют теги из --> [ tags_registry.yaml ]
      ^                                                                                           ^
      | использует                                                                                | загружается
      +-----------------------------------------<------------------------------------------ [ TagRegistryService ]
```

### Часть 2: Инструкции по расширению мира

Вот пошаговые руководства, как безопасно и правильно добавлять в игру новый контент.

#### Руководство №1: Как внедрить новый ТЕГ

Предположим, мы хотим добавить тег **`atmosphere:spore-filled`** (атмосфера, наполненная спорами).

**Шаг 1: Объявить "ЗАКОН" — `data/tags_registry.yaml`**
Это самый важный шаг. Откройте `tags_registry.yaml` и добавьте новый тег в подходящую категорию. Например, в `defining_tags`.

```yaml
# data/tags_registry.yaml

defining_tags:
  # ... другие категории
  atmosphere_attributes: # Новая или существующая категория
    category: physical
    tags:
      atmosphere:spore-filled: # ID нашего нового тега
        name: "Наполненная спорами"
        description: "Воздух в этой области насыщен спорами, что может влиять на видимость и дыхание."
  # ...
```

**Шаг 2 (Опционально): Определить Глобальные Правила**
Если новый тег должен хорошо или плохо сочетаться с другими, добавьте правило в `global_compatibility_rules` в том же файле. Например, пусть он плохо сочетается с сильным ветром.

```yaml
# data/tags_registry.yaml

global_compatibility_rules:
  # ...
  conflicts:
    - tags: ["atmosphere:spore-filled", "anomaly:breathing"] # Наш новый тег и тег сильного ветра
      penalty: 0.5
      reason: "Сильный ветер развеивает споры"
  # ...
```

**Шаг 3: Применить тег к Сущностям — `data/data_tables/*.yaml`**
Теперь, когда тег "легализован", присвойте его нужным биомам, расам или предметам. Например, добавим его в "Споровую саванну".

```yaml
# data/data_tables/anatomy.yaml

BIOMES:
  - id: spore_savanna
    name: "Споровая саванна"
    # ...
    defining_tags:
      - resource:dream_spores
      - danger:environmental
      - atmosphere:spore-filled  # <--- ВОТ ОН!
    # ...
```

**Готово!** Теперь ваша система знает о новом теге, и `CompatibilityService` будет автоматически его учитывать при расчетах.

#### Руководство №2: Как внедрить новую ЛОКАЦИЮ (Биом)

Предположим, мы хотим добавить новый биом — **"Кристаллические сады" (`crystal_gardens`)**.

**Шаг 1: Создать "статью в энциклопедии" — `data/data_tables/anatomy.yaml`**
Откройте файл и добавьте новую запись в список `BIOMES`. Заполните все ключевые поля.

```yaml
# data/data_tables/anatomy.yaml

BIOMES:
  # ... другие биомы
  - id: crystal_gardens # 1. Уникальный ID
    name: "Кристаллические сады"
    description: "Тихая пещера, где из земли растут хрупкие, резонирующие кристаллы, похожие на растения."
    
    # 2. Определяем его суть через теги из реестра
    pillar_tags:
      terrain_type: terrain:subterranean
      surface_stability: surface:stable
      fertility: ecology:unique
      
    defining_tags:
      - resource:resonance_crystal
      - anomaly:sound
      - location:sacred
      
    # 3. Правила совместимости
    compatibility:
      favored_races: [geodes, synaptics] # Какие расы здесь процветают
      disfavored_races: [ursines, kith]  # Какие избегают
      can_border: [luminous_fungi_caves, crystal_geode, silent_zone] # С кем может граничить
      cannot_border: [toxic_swamp, pulsating_plains] # С кем не может
      
    # 4. Контент
    content_generation:
      common_fauna: [crystal_cricket, resonating_moth] # Будущие ID из ecosystem.yaml
      common_flora: [echo_blossom, crystal_grass]
      resources:
        - id: resonance_crystal_raw
          abundance: abundant
      poi_types: # Типы POI (деревня, руины, логово), которые могут тут появиться
        - sacred_site
        - mining_camp
      poi_density: 2 # Сколько POI в среднем
      
    # 5. Параметры для генератора карты
    noise_generation:
      elevation: [0.1, 0.4]   # Предпочитает низкие высоты
      temperature: [0.2, 0.5] # Прохладный
      moisture: [0.3, 0.6]    # Умеренно сухой
      conductivity: [0.7, 1.0] # Высокая "магическая" проводимость
      
    spawn_weight: 4 # 6. Редкость (чем ниже, тем реже)
```

**Шаг 2: Разрешить появление биома в регионах**
Биом создан, но система не знает, в каких регионах он может появляться. Откройте `data/data_tables/anatomy.yaml` и добавьте `id` нового биома в `allowed_biomes` для подходящих `REGION_TYPES`. Например, для "Альвеолярных каверн".

```yaml
# data/data_tables/anatomy.yaml

REGION_TYPES:
  - id: alveoli_caves
    # ...
    biome_generation:
      allowed_biomes:
        - luminous_fungi_caves
        - wind_tunnels
        # ...
        - geode_ruins
        - crystal_gardens # <--- РАЗРЕШАЕМ ПОЯВЛЕНИЕ
      # ...
```

**Шаг 3: Создать шаблон для генерации названий — `data/data_tables/location_templates.yaml`**
Чтобы названия биомов были разнообразными, добавьте для него шаблон.

```yaml
# data/data_tables/location_templates.yaml

templates:
  # ...
  crystal_gardens_template: # ID должен совпадать: [biome_id]_template
    name_patterns: ["{Adjective} Сады", "Кристальный {Noun}", "Песня {Noun}"]
    adjectives: [Звенящие, Тихие, Резонирующие, Хрустальные]
    nouns: [Эха, Камня, Тишины]
```

**Шаг 4 (Рекомендуется): Запустить валидатор**
Откройте терминал в корне проекта и выполните команду:
```bash
python validators/yaml_validator.py
```
Скрипт проверит, не допустили ли вы опечатку в тегах и правильно ли все настроили.

Теперь ваша новая локация полностью интегрирована в систему и будет появляться при генерации мира, а все сервисы будут корректно понимать её свойства и правила взаимодействия.
